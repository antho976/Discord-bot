### **🏗️ Modular Architecture (Refactored from monolithic index.js)**

The original 42,866-line monolithic `index.js` was split into 10 files (81.2% reduction). The codebase uses a factory/dependency-injection pattern for modules.

#### Module Patterns

| Pattern | Used By | Description |
|---------|---------|-------------|
| **State Injection** `_getState()` | Render modules (analytics, config, community) | `initXxxTabs(getStateFn)` called at startup; each render function calls `_getState()` to get current state |
| **Route Registration** `registerXxx(app, deps)` | express-routes orchestrator, rpg-routes, smartbot-routes, features | Factory function receives Express `app` + destructured deps; routes run at request-time |
| **Sub-registrar** `registerXxxRoutes(app, deps)` | 6 sub-route files in `modules/routes/` | express-routes orchestrator delegates to domain-specific registrars (page, pets, twitch, config, leveling, events) |
| **Stream Factory** `registerStreamManager({ sv, ... })` | stream-manager.js | Factory receives `streamVars` object (`sv`) for mutable primitives + standard deps; returns exported functions |
| **Feature Factory** `registerFeatures(app, deps)` | features/index.js | Orchestrates 44 standalone modules, registers 50+ API routes + background tasks; returns hook functions for discord-events integration |
| **Mixin / Object.assign** | 12 RPG handler files in `rpg/commands/` | Each file exports a `{ method1, method2 }` object; core RPGCommand.js applies via `Object.assign(RPGCommand.prototype, handlers)` |
| **Event Registration** `registerXxx(deps)` | discord-events | Factory function registers all `client.on()` handlers with deps closure |
| **Pure Export** | rpg-editor-tab, rpg-tabs | Zero external deps, export render functions directly |

## 12. 🏆 Leveling System

### XP Sources
| Source | Amount | Cooldown |
|--------|--------|----------|
| Messages | 15-25 XP | 45 seconds |
| Voice | 5 XP/min | Continuous |
| Reactions | 2 XP | Per reaction |

### Level Formula
**Increment Mode (Default):**
```
XP for Level N = baseXp(100) * N + xpIncrement(50) * N*(N-1)/2
```
**Custom Mode:** Per-level XP requirements

### Prestige System
- **Thresholds:** [0, 500, 1500, 3500, 7000, 12000, 20000, 35000, 55000, 80000, 120000]
- **Ramp Zone:** 15 levels before threshold = 15% harder/level
- **Tier Multiplier:** `1.5^tier` XP scaling
- **Rewards:** Role assignment per prestige tier

### Weekly Tracking — [discord-events.js L4401-4403](modules/discord-events.js#L4401-L4403)
- Stored in `weeklyLeveling` object: `{ userId: { xp, lastGain } }`
- XP from messages, voice, and reactions all increment `weeklyLeveling[userId].xp`
- `/weekly` command filters entries where `lastGain >= 7 days ago` (sliding window)
- **⚠️ No actual reset:** `weeklyLeveling` accumulates indefinitely — old entries are never purged. The leaderboard uses a 7-day `lastGain` filter to simulate a weekly reset.

### XP Decay — [leveling-routes.js L147-149](modules/routes/leveling-routes.js#L147-L149)
- **Configurable via dashboard:** `enableXpDecay` (default: false), `xpDecayDays` (default: 14), `xpDecayPercent` (default: 5%)
- **⚠️ Not implemented at runtime:** The config is saved and displayed, but no interval or cron job actually applies decay. This is a stub — it configures but never executes.

### Features
- Global XP multiplier (0.5x-10x with expiration)
- Per-user XP multiplier with expiration
- Level milestones [10, 25, 50, 100]
- Role rewards per level
- Weekly tracking (separate from overall — see above)
- XP decay (configurable via dashboard — see above)
- Level-up channel (configurable or same channel)
- Custom level-up message with variables: {user}, {mention}, {level}, {prestige}, {xp}, {next_level}

## 27. 🔒 Security

### Express Middleware
| Header | Value |
|--------|-------|
| HSTS | Strict-Transport-Security |
| X-Content-Type | nosniff |
| X-Frame-Options | SAMEORIGIN |
| Trust Proxy | Level 1 (for Render) |

### Authentication & Session Security
- Session-based (cookie with `SameSite=Strict`, `HttpOnly`, `Secure`)
- Multi-account support with 4 tiers (owner/admin/moderator/viewer)
- Per-page access control via `requireAuth` + `requireTier()`
- **Password complexity:** 8+ characters, requires uppercase + lowercase + digit
- **Startup guard:** Bot refuses to start if `DASHBOARD_PASSWORD` env var is not set
- **Session persistence:** Sessions saved to `sessions.json` and restored on restart
- **Socket.IO auth:** WebSocket connections require valid session cookie

### API Security
- **API key comparison:** Uses `crypto.timingSafeEqual()` to prevent timing attacks
- **Endpoint protection:** All Twitch/admin POST endpoints require `requireAuth + requireTier('owner')`
- **Twitch token masking:** OAuth tokens are masked in HTML (only last 8 chars shown)
- **CORS:** Not configured — dashboard is **same-origin only**. No `cors` middleware, no `Access-Control-*` headers. External integrations must use API keys via server-side calls (curl/fetch), not browser requests.

### Rate Limiting — [index.js L2418-2443](index.js#L2418-L2443)
| Limiter | Scope | Limit | Window | Applied To |
|---------|-------|-------|--------|------------|
| **API rate limiter** | Per-user+path (or per-IP+path) | 30 req | 60s | `GET /api/v1/:resource` only — [index.js L4836](index.js#L4836) |
| **Login brute-force** | Per-IP | 5 attempts | 15 min | `POST /login` — [index.js L3120-3140](index.js#L3120-L3140) |

- API limiter key: `username:path` (or `ip:path` if unauthenticated). In-memory `Map`, cleaned every 5 min.
- **No `X-RateLimit-*` headers** are sent — only a `429 JSON` response on exceedance.
- **Twitch API tracking** ([index.js L1924-1933](index.js#L1924-L1933)): Monitors Twitch `ratelimit-remaining/limit/reset` headers for observability (not enforcement).

### Audit & Monitoring
- **Dashboard audit log:** All critical dashboard actions are tracked via `dashAudit()`
- **Structured logging:** `log(level, category, message, meta)` utility with configurable `LOG_LEVEL` env var
- **Global Express error handler:** Catches all unhandled route errors, logs and returns safe 500 response

### File Upload
- Images only (MIME check)
- 8MB max file size
- Stored in persistent directory

### Infrastructure
- **Graceful shutdown:** SIGTERM/SIGINT handlers save state + sessions before exit (10s timeout)
- **File write queue:** `saveJSON()` uses async write queue to prevent race conditions
- **Auto-backup:** Every 12 hours, copies all data/*.json to `data/backups/` (keeps last 7)

## 31. 🧠 SmartBot Conversation Pipeline

### Message Flow
```
messageCreate (discord-events.js L4151)
  → Pre-processing (ignore bots/DMs, sync leveling/stream data, inject userMemory)
  → smartBot.processMessage(msg, botUserId)   [smart-bot.js L8395]
    → A. Always-run training (every message, even no-reply)
    → B. Anti-spam guard (detectSpamBurst)
    → C. shouldReply() decision cascade
    → D. generateReply() waterfall
    → E. Post-processing (anti-repetition, anti-echo, adaptive length, typo sim)
  → Send reply (or reaction emoji)
```

### A. Training Pipeline (runs on every message)
| Step | Method | Description |
|------|--------|-------------|
| Topic detection | `detectTopics(content)` | Keyword matching against 55+ TOPICS map |
| Markov training | `markov.train(content, topic)` | Feeds bigrams/trigrams per topic |
| Word graph | `wordGraph.train(content)` | 3-word window associations (max 3000) |
| TF-IDF | `tfidf.train(content)` | Document frequency scoring |
| Channel memory | `memory.addMessage(...)` | Per-channel history (30 msgs max) |
| User prefs | `_trackUserPreferences(userId, content)` | Per-user topic affinities |
| Social tracker | detect reply chains + proximity | User interaction pairs |
| Trending | `trending.record(topic)` | 48h hourly + 14d daily buckets |
| Slang | `slangTracker.record(content, userId)` | Server expressions (2-4 word, max 200) |
| Auto-learning | `learnedKnowledge.recordMention(...)` | Learns subjects (3+ mentions, 2+ users) |
| Community opinions | `communityOpinions.record(subj, sentiment)` | Sentiment per subject |
| Expertise | `expertise.record(userId, topic, helpful)` | Topic experts (10+ msgs) |
| Feedback | `_detectFeedback(...)` | Checks reactions on bot's last reply |

### B. Anti-Spam (`detectSpamBurst`) — [smart-bot.js L1560](smart-bot.js#L1560)
Returns `true` if last 5 messages have: 5 in <10s with 4+ from same user, OR ≤2 unique contents (flood).

### C. Reply Decision — `shouldReply()` — [smart-bot.js L6920](smart-bot.js#L6920)

## 32. 🏗️ Multi-Server / Guild Switching

## 35. ⚔️ RPG Combat Mechanics

### Damage Formula
**Basic Attack:**
```
baseDamage = STR × 1.1
± variance (damageVariance + worldState bonus)
× stance.damageDealt (aggressive = 1.15, defensive = 0.9)
× offensive buffs
- (defense × defenseMultiplier)  [min 1]
× worldState modifier
× TYPE_CHART[attackElement][defenderType]
× stance.damageTaken (defender side)
- defensive effects (guard, shields)
```

**Skill-Based:**
```
damage = skill.damageCalc(stats, skillLevel)
× combatStyle modifiers × comboBonus × stance
× critMultiplier (if crit) × offensiveBuffs
- (defense × defenseMultiplier)
× worldState × TYPE_CHART × stance.damageTaken
```

**Critical hits:** Chance = `min(95%, DEX × 0.5 + 5%)`, damage = `×1.5`

### Type Chart — 10 Elements — [CombatSystem.js L17](Discord%20bot%20-%20test%20branch/rpg/systems/CombatSystem.js#L17)
Pokémon-style: `fire→grass = 2×`, `fire→water = 0.5×`, `holy↔shadow = 2×`, `electric→water = 2×`, `ice→grass = 2×`, etc.

### Stances — [CombatSystem.js L32](Discord%20bot%20-%20test%20branch/rpg/systems/CombatSystem.js#L32)
| Stance | Dmg Dealt | Dmg Taken | Accuracy | Crit | Dodge |
|--------|-----------|-----------|----------|------|-------|
| balanced | 1.0 | 1.0 | 0 | 0 | 0 |
| aggressive | 1.15 | 1.1 | 0 | 0 | 0 |
| defensive | 0.9 | 0.8 | 0 | 0 | 0 |
| focused | 1.0 | 1.0 | +0.1 | +0.1 | 0 |
| evasive | 1.0 | 1.0 | -0.1 | 0 | +0.1 |

### Combat Flow — [CombatSystem.js L338](Discord%20bot%20-%20test%20branch/rpg/systems/CombatSystem.js#L338)
1. Environment effects processed
2. DoT effects tick
3. Boss phase transition check (HP%)
4. **Player action:** AI selects best skill via `evaluateSkillUsage()`, or manual via `executeRoundWithSkill`
5. If enemy HP ≤ 0 → victory (or next enemy in multi-encounters)
6. **Enemy action:** 55% skill use, otherwise basic attack — skipped if stunned
7. If player HP ≤ 0 → defeat
8. Update cooldowns, prime enemy intent, round++

### Combo System — [skill-combos.js](Discord%20bot%20-%20test%20branch/rpg/data/skill-combos.js#L6)
Two-skill chains within 5 seconds. Examples:

| Class | Chain | Multiplier | Effect |
|-------|-------|------------|--------|
| Warrior | cleave → whirlwind | ×1.5 | bleed (3T, 20%/turn) |
| Warrior | parry → execute | ×1.6 | +20% accuracy |
| Mage | time_warp → overcharge_blast | **×2.0** | CD reduction |
| Mage | frost_nova → meteor | ×1.5 | stun 2T |
| Rogue | shadow_step → assassinate | ×1.6 | +30% accuracy |
| Paladin | divine_shield → martyr_strike | ×1.5 | shield bonus |

Detection: `COMBO_HISTORY` Map tracks `{lastSkill, lastTime}` per user. `recordSkillUsage()` checks combo table.

### Boss Phases — [boss-abilities.js](Discord%20bot%20-%20test%20branch/rpg/data/boss-abilities.js#L6)
3-phase system keyed on HP%:
- **Phase 1:** 100% HP → pool of 3 abilities
- **Phase 2:** 50-65% HP → harder abilities
- **Phase 3:** 25-30% HP → ultimate abilities

Examples: `inferno_lord` (100/60/30%, weak=water, resist=fire), `frost_queen` (100/50/25%, weak=fire), `shadow_king` (100/55/25%, weak=holy).

## 36. 🏰 RPG Procedural Generation

### Roguelike Dungeon — [RoguelikeManager.js](Discord%20bot%20-%20test%20branch/rpg/systems/RoguelikeManager.js)
**Structure:** 26 floors, 5 sections of 5 floors + final boss at floor 26. Stats reset to base level 1 each run.

**Floor generation** (3-4 rooms per floor):
- Weighted random from 12 room types: skill(20), treasure(15), healing(15), combat(12), rest(10), shop(10), trap(10), mystery(8), elite(5), + 3 unlockable
- No duplicate types per floor
- 40% chance room modifier, 30% chance random event
- 25% per floor: special challenge (noHealing, doubleEnemies, damageReduction, timeLimit)

**Room modifiers:** blessed (+25% rewards), cursed (-30%), flooded (-50% healing), frozen (-20% dmg), inferno (+20% dmg + HP loss), thunderstorm (30% chain hit)

**Currencies:** Gold (💰), Essence (✨, 50-75 to unlock rooms), Void Shards (🔷, boss drops)

**Scaling:** `healthPerBoss = ×1.15`, `damagePerBoss = ×1.12` per section

**Status effects (9):** Regeneration (10% HP/room), Poisoned (15% HP/room), DamageBoost (+15% for 2 rooms), etc. Consumables: potions, antidote, cleanse scroll, phoenix feather.

### Tower System — [TowerManager.js](Discord%20bot%20-%20test%20branch/rpg/systems/TowerManager.js)
**Structure:** 100 floors, boss every 5th floor, requires Level 150+. "Niflheim" ice theme.

**Scaling:**
| Type | HP | DMG | DEF |
|------|----|-----|-----|
| Regular | `10,000 × 1.08^(floor-1)` | `500 × 1.06^(floor-1)` | `150 × 1.05^(floor-1)` |
| Boss | `50,000 × 1.12^(floor/5-1)` | `1,200 × 1.10^(floor/5-1)` | `300 × 1.08^(floor/5-1)` |

**Rewards:** gold `×(1 + floor×0.15)`, essence `×(1 + floor×0.10)`, materials `floor(1 + floor×0.08)`

Floor 100 boss: "Heart of Niflheim" ❄️ — gold/essence ×5, 5 legendary lootboxes, T6 enchants/potions. 20 themed bosses at milestone floors. Legendary chance: `+0.1%/floor` (max +15%), boss floors `+15%`.

## 38. 💾 Data Schemas & Persistence

### State Management Model
```
state.json (master runtime store) ← loadState() / saveState() [debounced 5s]
data/*.json (domain files)        ← loadJSON(path, default) / saveJSON(path, data) [per-file write queue]
config.json (root + data/)        ← vestigial; runtime config lives inside state.config
```

### Key File Schemas

**[state.json](state.json)** — Master runtime state:
```json
{
  "isLive": true, "lastStreamId": "317272662105",
  "stats": { "totalStreams": 1, "totalViewers": 50, "peakViewers": 50 },
  "history": [{ "startedAt": "...", "viewers": 50, "game": "IdleOn" }],
  "apiHealth": { "failures": 0, "lastSuccess": 1769400325574 },
  "schedule": {...}, "leveling": {...}, "levelingConfig": {...},
  "giveaways": [], "polls": [], "reminders": [],
  "notificationFilters": [], "customCommands": [],
  "config": { "ROLE_ID": null, "notificationRoles": {}, "notificationChannels": {} }
}
```

**[data/accounts.json](data/accounts.json)** — Dashboard auth:
```json
{
  "accounts": [{
    "id": "uuid", "username": "admin",
    "password": "salt:hash",  // PBKDF2
    "tier": "owner",          // owner | admin | moderator | viewer
    "createdAt": 1771714464887, "lastLogin": 1771715281688
  }],
  "initialized": true
}
```

**[data/players.json](data/players.json)** — RPG player data (array):
```json
[{
  "userId": "247868119605968898", "username": "player1",
  "level": 102, "xp": 8769151, "xpToNextLevel": 10477033,
  "gold": 9965870, "class": null,
  "narrativeChoice": "tactician", "internalClass": "mage",
  "skills": ["fireball", "ice_spike"],
  "skillCooldowns": {}, "inventory": [], "equipment": {}
}]
```

### Configuration Precedence
- `config.json` (root) and `data/config.json` are both **vestigial** — effectively unused
- Runtime config lives in `state.config` inside `state.json`
- `saveConfig()` is just an alias for `saveState()`
- All domain data (`moderation.json`, `api-keys.json`, `pets.json`, etc.) is independent via `loadJSON`/`saveJSON`

### Corruption Handling
- `loadState()`: wraps `JSON.parse` in `try/catch` → returns deep copy of `defaultState` on failure
- `loadJSON(path, default)`: `try/catch` → returns `default` on parse error
- `saveJSON()`: async write queue per file path → prevents race conditions

## 39. 🔑 API Key Management

### Dashboard API Keys — [index.js L4811](index.js#L4811)

**Storage:** [data/api-keys.json](data/api-keys.json) — `{"keys": [...]}`

**Key Schema:**
```json
{
  "id": "uuid", "name": "My Integration",
  "key": "dk_<48 hex chars>",  // crypto.randomBytes(24)
  "permissions": ["read"],
  "createdAt": 1771714464887, "lastUsed": null, "uses": 0
}
```

**CRUD Routes:**
| Route | Auth | Purpose |
|-------|------|---------|
| GET `/api/api-keys` | owner | List keys (truncated: first 8 chars + `...`) |
| POST `/api/api-keys/create` | owner | Generate `dk_` + 24 random bytes hex |
| POST `/api/api-keys/delete` | owner | Remove by `id` |

**Public API:** GET `/api/v1/:resource` — authenticated via `x-api-key` header or `?key=` query param, validated with `crypto.timingSafeEqual()`.

Available resources: `stats`, `members`, `commands`, `growth`.

**Separate from env API keys:** External service keys (Groq, Weather, etc.) are loaded from environment variables and passed to SmartBot via `smartBot.setApiKeys()` at startup.

## 40. 🚀 Development & Deployment

### Local Development
1. `npm install`
2. Create `.env` with minimum: `DISCORD_TOKEN`, `DASHBOARD_PASSWORD`
3. Optional: `TWITCH_CLIENT_ID`, `TWITCH_CLIENT_SECRET`, `BROADCASTER_ID`, `GROQ_API_KEY`, etc.
4. `npm start` (or `node index.js`)
5. Without `PERSISTENT_DATA_DIR`, data reads from `./data/` and state from `./state.json`

No build step, no hot-reload, no dev mode. ESM modules (`"type": "module"`).

### Production — [render.yaml](render.yaml)
```yaml
services:
  - type: web
    name: discord-bot
    env: node
    buildCommand: npm install
    startCommand: node --max-old-space-size=6144 index.js
    healthCheckPath: /healthz
    disk:
      name: bot-data
      mountPath: /opt/render/data
      sizeGB: 1
```

### Startup Sequence
1. Load env vars, initialize state, create Express app
2. Register all modules (stream-manager, express-routes, discord-events, etc.)
3. Start HTTP server on `PORT` (default 10000)
4. Discord client login
5. `ready` event: cache members, register slash commands, start all intervals

### Shutdown
`SIGTERM`/`SIGINT` → save state + sessions → close HTTP server → destroy Discord client (10s force-kill timeout).

## 41. 🎨 Known Issues & Limitations

### Architecture
| Issue | Impact |
|-------|--------|
| No database — all JSON files | Risk of corruption under concurrent writes; no query capability; entire file read/written on each operation |
| Rate limiting in-memory only | Lost on restart; no shared state across instances |
| Sessions in-memory (Map) | Lost on restart (mitigated by `sessions.json` save/restore, but still single-process) |
| No CORS configuration | Dashboard is same-origin only; external integrations must use API keys |
| `state.json` is a god object | Single file holds stream, leveling, giveaways, polls, reminders, config, and 30+ other domain states |

### Operational
| Issue | Impact |
|-------|--------|
| Data not isolated per guild | Multi-server mode shares all JSON files globally |
| No database migrations | Schema changes require manual JSON transformation |
| No CI/CD pipeline | No automated tests on push, no staging environment |
| Line-number links in docs | Break when code changes; would need function-name anchors |
| Render free tier limitations | 512MB actual RAM, 6GB swap; spin-down after inactivity |

### Content/Features
| Issue | Impact |
|-------|--------|
| `config.json` (root + data/) vestigial | Two dead config files; runtime config lives in `state.config` |
| XP decay undocumented trigger | 5%/14 days mentioned but exact cron/check interval unclear |
| Weekly XP reset timing | Not explicitly documented when the reset runs |

## 44. ⚠️ Error Handling Strategy

### Global Process Handlers — [index.js L467-476](index.js#L467-L476)
| Handler | Behavior |
|---------|----------|
| `process.on('unhandledRejection')` | Logs via `addLog()`, **does not exit** — bot keeps running |
| `process.on('uncaughtException')` | Logs, calls `saveState()`, then `process.exit(1)`. Both wrapped in try/catch to prevent crash-in-crash |

### Express Global Error Handler — [index.js L5638-5642](index.js#L5638-L5642)
```js
app.use((err, _req, res, _next) => {
  console.error(err);
  addLog('error', ...);
  res.status(500).json({ success: false, error: 'Internal server error' });
});
```
- Returns safe generic 500 — never leaks internal details to client.

### try/catch Patterns (~30+ blocks in index.js)
| Pattern | Examples |
|---------|----------|
| JSON file loading with silent fallback | `loadJSON()` at L116, L379, L1254, L3813 |
| Data initialization (pets, state) | L200-265 |
| State/log persistence | L469, L473-474 |
| Discord audit log operations | L442-458 |
| Network/API calls (Twitch, Groq) | L486, L509, L551, L830 |
| Guild/Server operations | L1300, L1325, L3213 |

**Philosophy:** "Fail silently, keep running" — most catch blocks are empty (`catch {}`) to prevent cascading failures. Only `uncaughtException` forces an exit.

## 46. 📝 Custom Commands System

### Creation
- **Slash command** `/addcommand` — [discord-events.js L3350-3362](modules/discord-events.js#L3350-L3362): Takes `name` + `response`
- **REST API** `POST /customcmd/add` — [events-routes.js L220-253](modules/routes/events-routes.js#L220-L253): Supports additional fields (`imageUrl`, `autoDeleteAfterUses`, `cooldownMs`, `allowedRoleIds`, `allowedChannelIds`, `category`, `tags`, `sendAsEmbed`), can create multiple at once

### Storage
Custom commands live in the `customCommands` array in memory, persisted via `saveState()` → `state.json`.

### Execution — [discord-events.js L4295-4360](modules/discord-events.js#L4295-L4360)
1. Message must start with `!`
2. Commands sorted by **name length descending** (phrases match before single words)
3. Checks: `paused`, `isCommandAllowed` (roles/channels), `cooldownMs`
4. Increments `uses`, records last 10 usage entries
5. Calls `formatCommandResponses()` → `sendCommandResponse()`
6. Auto-deletes if `autoDeleteAfterUses` threshold reached

### Placeholders — [discord-events.js L4018-4100](modules/discord-events.js#L4018-L4100)
| Placeholder | Value |
|-------------|-------|
| `{user}` | Username |
| `{mention}` | `<@id>` mention |
| `{displayname}` | Server display name |
| `{server}` | Server name |
| `{channel}` | `<#id>` channel mention |
| `{members}` | Member count |
| `{date}` | Local date |
| `{time}` | Local time |
| `{level}` | User's leveling level |
| `{xp}` | User's XP |
| `{uses}` | Command use count |
| `{args}` | Text after command |
| `{userid}` | User ID |
| `{random:x,y,z}` | Random choice from list |
| `{pick:1-100}` | Random number in range |

### Constraints
- No explicit limit on number of commands
- Usage history capped at 10 entries per command
- Per-user cooldown via `cooldownMs`
- Role/channel restrictions via `allowedRoleIds`, `allowedChannelIds`
- Auto-delete after N uses (`autoDeleteAfterUses`)
- Response sent as embed by default; supports image URLs

## 52. � Security Hardening — Implemented

### Security Module — [modules/security.js](modules/security.js)
Centralized security middleware module providing 18 security functions across 7 categories:

| Category | Functions | Description |
|----------|-----------|-------------|
| **Headers** | `setupHelmet()` | `helmet@8.1.0` — CSP, HSTS (1yr), X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy, remove X-Powered-By |

| **Files** | `writeSecureFile()` | Write files with `0o600` permissions (owner read/write only) |

### CSRF Protection Flow
```
1. User loads dashboard page (GET)
2. Server generates _csrf token → sets cookie + X-CSRF-Token header
3. dashboard-shell.js intercepts all fetch() calls
4. On POST/PUT/DELETE: reads _csrf cookie → adds X-CSRF-Token header
5. Server validates: header token must match stored token for session
6. Mismatch → 403 "Invalid CSRF token"
```
Frontend integration: [public/dashboard-shell.js](public/dashboard-shell.js) — auto-injects CSRF token via `fetch()` wrapper.

### Discord OAuth Flow
```
1. User clicks "Login with Discord" on /login
2. GET /auth/discord → generates state token (IP-bound) → redirect to Discord
3. User authorizes → Discord redirects to /auth/discord/callback?code=...&state=...
4. Server validates state (CSRF + IP + 5min TTL)
5. Server exchanges code for access_token (server-to-server POST)
6. Server fetches /users/@me with access_token → gets discordId, username, avatar
7. Server immediately revokes access_token (no need to store it)
8. If discordId exists in accounts → log in directly
9. If new → create viewer account → log in
10. Session cookie set → redirect /select-server
```

## 56c. 🎯 SmartBot Training Effectiveness Overhaul

Major fixes to make training actually improve bot behavior, reduce repetitive/stale output, and ensure the feedback loop works.

### Critical Bug Fix: Training Feedback Was Going Nowhere
| ID | Fix | Details |
|----|-----|---------|
| T1 | Feedback key mismatch | Training recorded feedback under `training:scenarioText` keys, but templates use `topic:replyText` keys. `recordFeedback()` only updates **existing** entries — so training feedback was silently discarded. Now records against the ACTUAL template key the bot used during generation (via `lastBotReplyDetails`) |
| T2 | Reply-based feedback | Additionally records feedback under `reply:replyText` keys so the same bad reply text gets penalized regardless of which template produced it |
| T3 | Batch feedback routing | Batch training items now pass their `_batchIdx` so feedback routes to the correct per-item template key (each batch item uses a unique training channel) |

### Batch Repetition Fixes
| ID | Fix | Details |
|----|-----|---------|
| T4 | Scenario deduplication | Batch generation now shuffles `TRAINING_SCENARIOS` and tracks used scenarios to avoid repeats |
| T5 | Reply deduplication | Retries generation up to 3 times if the reply text matches a previous item in the batch |
| T6 | Per-item training channels | Each batch item uses `training-channel-{i}` instead of shared `training-channel`, preventing cooldown/dedup interference between items |

### Training Effectiveness Improvements
| ID | Fix | Details |
|----|-----|---------|
| T7 | Fuzzy shouldReply boost | `shouldReply()` now uses the inverted pair index for fuzzy matching — if 2+ trained pairs share words with the message, reply chance boosted 4× (was only exact match at 6×) |
| T8 | Expanded scenario pool | TRAINING_SCENARIOS expanded from ~40 to ~75 items covering: news/patches, social/LFG, short reactions, meta questions, new member greetings |

### Staleness & Close Match (from previous session)
| ID | Fix | Details |
|----|-----|---------|
| T9 | Auto-generate next batch | When all 5 batch items are rated and submitted, a new batch auto-generates after 400ms |
| T10 | 🤏 Close Match rating | New rating option in single and batch mode — saves pair with half score (0.5), flags as needing improvement |
| T11 | Staleness decay | Pairs not updated in >30 days get extra 2%/day decay. Fuzzy matching penalizes stale pairs: -5% at 14d, -15% at 30d. Visual ⚠️ indicators in pairs library |
